Measuring Provider-Side Linkage Risk in LLM Agent Traffic}
\author{Anonymous Submission}

\begin{abstract}
Enterprise LLM gateways can strip account and session identifiers before forwarding agent requests,
creating an apparent anonymity boundary. We show that plaintext alone can undermine this boundary
for a passive provider. Recurring task objects and replayed context let otherwise unlabeled calls form
pseudonymous workflows and, when handles persist, longitudinal watchlists.

We define provider-side linkage risk and a controlled measurement design that separates
provider-visible traffic from linkage truth hidden from the attack. On software-engineering and
business-tool traces, paired tests perturb paths, replayed context, stable handles, and timing. These
controls isolate three channels: direct content exposure, workflow continuity, and longitudinal
propagation. We instantiate them with two bounded linkage pipelines: \carp for sparse discovery and
\asl for selective linkage from context evolution, tool transitions, and typed handles.

Direct workspace and business-object handles explain the easiest recovery, but they are not the only
signals. After paths, repository slugs, and internal domains are removed, structured content links
unseen OpenHands projects at F1 0.682. On non-code traces, time-independent continuity remains at F1
0.843 after intents are rephrased. \asl more than triples \carp's incremental-context F1 on
Open-SWE, improves historical tool-agent linkage without false session assertions, and maintains a
bounded active working set through 500K requests. Conversely, combining handle abstraction with
non-cumulative requests nearly eliminates software-session linkage, showing that the channels need
separate controls.
Identifier stripping is therefore not content de-identification: agent privacy boundaries must cover
persistent task objects, replayed context, and provider log retention.
\end{abstract}

\section{Introduction}

An enterprise gateway can strip every account and session identifier from an LLM-agent call yet
still forward the content that makes the call recognizable. Paths name repositories, tool arguments
name business objects, and later turns often replay earlier messages. The calls are unidentified at
the protocol layer but may remain linkable in content.

An API broker or gateway is optional middleware between an agent and a model provider; it can
authenticate callers, route models, meter usage, and keep local state. In \emph{protocol-level
identifier stripping}, it keeps end-customer account, organization, trace, and session IDs local;
the provider authenticates the broker but receives no underlying caller mapping. Plaintext messages,
tool schemas, and outputs still pass for inference. Direct agent-to-provider deployments skip this
step and may expose still more protocol identifiers. We study the residual risk in the more protective
brokered setting.

Agent traffic makes the gap consequential. Repeated repository namespaces in code agents and
account, order, queue, or tenant objects in tool agents act as quasi-identifiers. A stable
pseudonymous cluster combines otherwise separated fragments into a
longitudinal view of co-evolving repositories and services, dependencies, failures, and migrations.
This exposes product direction, operational relationships, and security architecture as commercial
or security intelligence, while a watchlist makes the observation repeatable.

Across domains, two mechanisms produce this view: \emph{persistent content handles} retain a
namespace or object, while \emph{context replay} retransmits prior messages or state. Paired controls
must separate them because removing one does not test the other.

\paragraph{Definition (provider-side linkage risk).}
For provider-visible requests $X=(x_1,\ldots,x_n)$ and a hidden relation $Y^\ell_{ij}$ indicating
whether requests $i,j$ share workflow or entity level $\ell$, let $S$ be a linkage score and
$\mathcal A_0$ attacks that ignore $X$ but may preserve the cluster-size prior. For attacker class
$\mathcal A$, define
\begin{equation}
\mathcal R_\ell=\sup_{A\in\mathcal A}\mathbb E[S(A(X),Y^\ell)]-
\sup_{A_0\in\mathcal A_0}\mathbb E[S(A_0,Y^\ell)].
\end{equation}
We instantiate $S$ with pairwise precision/recall/F1 and later assignment, and separately measure
evidence accumulated within predicted components. A \emph{linkage channel} is a provider-visible
signal family $c$ whose controlled transformation $T_c$ reduces $\mathcal R_\ell$ while truth stays fixed.

Prior privacy attacks recover training examples, membership, or attributes from learned models
\cite{shokri2017membership,carlini2021extracting}. Record linkage, fingerprinting, and modern
unsupervised text attribution show that removing direct identifiers need not prevent association
\cite{fellegi1969theory,sweeney2002kanonymity,narayanan2009anonymizing,eckersley2010unique,bertocco2023unsupervised}.
Our setting places record linkage inside normal model inference: the model API provider observes a
live request stream and reconstructs pseudonymous cross-request partitions from a cold start.

The central measurement challenge is attribution: successful linkage may reflect a directly readable
field, retransmitted history, semantic continuity, or a stable object recurring across workflows. We
therefore ask:

\begin{quote}
Which linkage channels remain after protocol identifier stripping, under what transformations do
they survive, and can a provider measure them under a bounded candidate budget?
\end{quote}

We make three contributions:
\begin{itemize}
    \item We identify provider-side linkage as a distinct privacy failure of protocol identifier
    stripping and decompose it into direct exposure, workflow continuity, and longitudinal propagation.
    \item We develop a controlled measurement protocol that separates provider-visible traffic from
    linkage truth and uses paired removal, rephrasing, concurrency, rotation, and collision tests to
    attribute risk. These controls reveal persistent content handles and context replay as cross-domain
    root mechanisms.
    \item We develop \emph{Cache-Anchored Rarity Percolation} (\textsc{CARP}) for budgeted sparse
    discovery and \emph{Agent-State Linkage} (\textsc{ASL}) for selective linkage using bounded state
    replay, tool/resource transitions, typed handles, negative evidence, and abstention. Across code
    and business-tool traces, we quantify residual linkage, transfer, stability-dependent watchlists,
    partial profiling, uncertainty, and cost.
\end{itemize}

\section{Related Work}

\paragraph{Record linkage, matching, and attribution.}
Statistical record linkage formalizes noisy matching \cite{fellegi1969theory}; modern systems combine
blocking and filtering with learned matchers
\cite{christen2012survey,papadakis2020blocking,li2020ditto}, and recent benchmarks quantify their
candidate-quality tradeoffs \cite{neuhof2024benchmark}. Collective entity resolution propagates
relational evidence \cite{bhattacharya2007collective}. \carp turns these primitives into a bounded
multi-level pipeline for nested agent requests, while \asl adds directed state evolution,
tool/resource causality, typed conflicts, and selective component growth. Quasi-identifiers, graph
and website de-anonymization, and author identification show why deleting names does not guarantee
unlinkability
\cite{sweeney2002kanonymity,narayanan2009anonymizing,panchenko2016website,narayanan2012author}.
Recent unsupervised attribution and LLM-based authorship-privacy studies extend that lesson to modern
text representations \cite{bertocco2023unsupervised,nguyen2025authorship}.

\paragraph{Language-model privacy.}
Membership inference and training-data extraction target information encoded in model parameters
\cite{shokri2017membership,carlini2021extracting}; LLMs can also infer sensitive attributes from
ordinary text \cite{staab2024beyond}, and text embeddings can be inverted to recover much of their
source content \cite{morris2023embeddings}. Recent text anonymization instead rewrites content while
balancing privacy and utility \cite{yang2025anonymization}. Differential privacy provides a
training-time guarantee \cite{dwork2006calibrating}; our setting concerns plaintext inference inputs
and the associations that emerge across repeated requests.

\paragraph{Agent evaluation and security.}
SWE-bench and OpenHands evaluate software problem solving and agent interfaces
\cite{jimenez2023swebench,wang2024openhands}; tau-bench evaluates tool-agent-user interaction
\cite{yao2024taubench}. Prompt-injection research shows that tool-integrated applications enlarge the
attack surface \cite{greshake2023indirect,debenedetti2024agentdojo,yi2025promptinjection}. These works
focus on task success or active compromise. We study passive linkage by the model provider.

Together, these literatures establish that content can remain identifying, but they do not model the
provider's cold-start problem of reconstructing multi-level Agent workflows from nested, evolving API
requests under a bounded candidate budget. Our measurement design and Agent-state linkage methods
address this gap.

\section{Threat Model and Problem}

\subsection{Parties and Visibility}

The request flow is agent $\rightarrow$ broker $\rightarrow$ model provider. The broker retains local
identities; the honest-but-curious provider receives the transformed payload described above.
Table~\ref{tab:assumptions} states its capabilities. The provider begins cold-start, with no target
registry or labeled history, and may derive a pseudonymous watchlist from earlier traffic. Such a
longitudinal view can support analytics, abuse investigation, or commercial and security
intelligence.

The broker terminates the client connection and creates a new upstream request, so the provider sees
the broker's upstream connection together with the forwarded inference content.

We separate the provider-visible view from sealed linkage truth. Test-instance decisions use only
provider observations; disjoint development labels select calibration settings, and held-out truth
is opened only for scoring.

This visibility contract occurs in configurable services: Cloudflare documents prompt/response
gateway logs \cite{cloudflare2026gatewaylogs}; OpenAI distinguishes abuse-log retention from
zero-data-retention controls \cite{openai2026datacontrols}; Microsoft distinguishes prompt
processing, stateful storage, and modified abuse monitoring \cite{microsoft2026foundryprivacy}.

\subsection{Reconstruction Tasks}

We distinguish direct exposure from cross-request reconstruction. \emph{Continuity} groups the
requests of one workflow; \emph{propagation} connects later traffic to an earlier component; and
\emph{profiling} aggregates attributes supported by the linked requests. Retaining early components
for repeated later assignment yields a pseudonymous \emph{watchlist}.

We map each dataset into four evaluation slots---session, user/customer, project/process, and
tenant/owner. In Open-SWE, project and owner denote repository and GitHub-owner truth; the
controlled tau-bench overlay supplies customer, project, and tenant entities.

We report pairwise precision/recall/F1 and purity. For project and organization scores, we exclude
within-workflow pairs so session grouping earns no longer-lived linkage credit. We score turn
ordering, later assignment, and field-level profiling separately, and we resample workflows,
entities, and organizations respectively.

Pairwise accuracy is misleading: only 0.083\% of the 6.48 million held-out Open-SWE pairs are
positive, so predicting no links is 99.917\% accurate with zero recall. Precision measures
contamination among asserted links; recall measures aggregation reach. A false positive can pollute
a profile or watchlist, whereas a false negative understates provider reach. We therefore report both
and sweep their cost ratio rather than treating F1 as a universal privacy loss.

\paragraph{Measurement requirements.}
A valid test seals truth from the attack, prevents within-workflow inflation, scores watchlists on
later traffic, and bounds candidate generation. \carp instantiates these requirements.

\paragraph{Observation-equivalence limit.}
Linkage is impossible without distinguishing information. If $k$ equally likely entities each
contribute $m$ requests and their \emph{complete} visible sequences are exchangeable, closed-set
entity accuracy is at most $1/k$. A predicted pair is then correct with probability at most
$p_k=k\binom{m}{2}/\binom{km}{2}=(m-1)/(km-1)$, giving expected pairwise F1 at most
$2(m-1)/(m(k+1)-2)$; merging the class attains the bound. Shared prompts or schemas alone do not
suffice if task text, handles, timing, or replay differ. The proof and controlled validation are in
the supplement.

\section{Measurement Design and Agent-Native Linkage}

Our design separates two questions that are often conflated: which visible channel carries a link,
and how an attacker can exploit that channel at scale. Paired transformations answer the first;
\carp and \asl answer the second through complementary sparse and state-aware linkage paths
(Figure~\ref{fig:carp}).

\subsection{Provider-Visible Evidence}

The attack view contains two intuitive signal families. \emph{Persistent handles} are repeated
namespaces or task objects such as repository names, paths, accounts, orders, queues, and tenants.
\emph{Continuity signals} indicate that one request grows from another, through replayed messages,
tool outputs, time, and length. We extract bounded lexical shingles, typed handles, tool/system
fingerprints, timing, length, and optional cache-like blocks. A structured parser reads tool records
and explicit identifiers; shared flight/product resources remain candidate evidence rather than
direct identity edges. The supplement and artifact report exact feature caps.

\subsection{Budgeted Linkage Pipeline}

\carp maps provider-visible streams to level-specific partitions under a per-request candidate
budget $B$. Five stages implement one principle: use cheap evidence to reduce the candidate set
before applying stronger evidence. Its name summarizes the
procedure: start from cache-like blocks, index rare recurring handles, then propagate only
unambiguous components.
\begin{enumerate}
    \item \emph{Block locally}: group by optional cache-like metadata; missing metadata forms one
    fallback block, which is the path used by Open-SWE.
    \item \emph{Index rare handles}: use bounded inverted indexes over trace-, user-, project-, and
    tenant-like values.
    \item \emph{Detect context growth}: propose session edges when a shorter request is substantially
    contained in a later request.
    \item \emph{Refine locally}: combine multiple weak similarities within a fixed pair budget.
    \item \emph{Propagate typed handles}: connect unambiguous user, process/project, or tenant
    components across blocks, while keeping shared resources candidate-only.
\end{enumerate}
Separate components prevent a user signal from silently becoming a project or organization link.
Local work is $O(nB)$, and cross-block propagation operates over extracted anchors rather than all
request pairs. We select thresholds on disjoint development data, freeze them for evaluation, and
stress-test them in Section~5; the supplement reports the complete parameter table.

\subsection{Agent-State Linkage}

\asl models each request as a point in an evolving Agent state. Its fixed-cap representation contains
hashes of replayable message events,
initial task and recent context, tool observations and action types, resource roots, and level-typed
handles. A one-pass index proposes predecessors from
these views inside an active time window. A link is accepted only when several independent evidence
families agree, no strong-handle or ordering conflict applies, and the best candidate clears a
runner-up margin; otherwise \asl abstains. Workflow components form first, and only same-level typed
evidence may subsequently connect user, project, or tenant components. Bounded indices and an active
state window cap retrieval work, while the output partition state grows linearly with the stream.
Exact settings and evidence-family ablations are confined to the supplement.

\subsection{Content Reconstruction and Longitudinal Assignment}

Within a predicted cluster, context containment and request time reconstruct the workflow sequence.
A structured profiler aggregates code, dependency, command, repository, service, and domain
evidence, retaining the supporting requests for every inferred attribute. A semantic counterpart
maps concise technical spans to a fixed ontology while filtering negation and migration contexts
\cite{reimers2019sentencebert,wang2020minilm}.

After sparse discovery, a content stage links rephrased non-code intents and path-abstracted project
sessions. It combines identifier-masked semantic similarity with typed-object and structural
agreement, allowing the experiments to distinguish literal replay from deeper task continuity.

For longitudinal tracking, early predicted clusters yield an entity watchlist. Exact typed anchors
map later requests back to an earlier cluster only when the anchor is unique or wins an
unambiguous vote. We use evaluation truth only to score these assignments.

\section{Datasets and Experimental Setup}

The evaluation layers evidence by the truth each source can support. Software-agent traces expose
workflow and repository structure, tool-agent traces expose cumulative dialogue and business
objects, and trace-grounded overlays supply controlled cross-workflow hierarchies. Together they
separate direct exposure, continuity, and longitudinal propagation without asking one dataset to
stand in for every deployment.

\subsection{Data Sources}

\paragraph{Open-SWE-Traces.}
Our main real-trace substrate is the \texttt{nvidia/Open-SWE-Traces} dataset
\cite{nvidiaopenswetraces}, containing trajectories
from software-engineering agent scaffolds including OpenHands
\cite{wang2024openhands}. We reconstruct provider-visible requests at standardized trajectory
checkpoints.
Evaluation uses workflow, repository/project, and GitHub-owner labels under a fixed development/test
split and a common set of trajectory checkpoints. Controlled provider-view timestamps support
temporal stress tests, while a separate SWE-agent sample tests cross-scaffold transfer.
OpenHands paths expose canonical repository and owner strings. We evaluate this direct exposure
separately from linkage after path abstraction.

\paragraph{tau-bench historical.}
tau-bench contains airline and retail tool-agent trajectories
\cite{yao2024taubench}. We convert each assistant turn and its cumulative preceding dialogue/tool
state into a provider-view request. A domain-stratified workflow split separates calibration from
evaluation, exposing both cumulative dialogue continuity and recurring business objects.

\paragraph{Trace-grounded overlays.}
Two Open-SWE overlays inject controlled user/environment/profile signals into real traces, either
retaining distinguishable user signals or intentionally sharing them. A controlled tau-bench entity
overlay places a synthetic tenant/customer/project hierarchy over
real trajectories and interleaves their requests in time.

Repository and owner handles are pervasive in Open-SWE, whereas tau-bench exposes identity and
process handles less uniformly and contains many ambiguous shared resources. This contrast lets us
test both direct namespace leakage and conservative association under object reuse.

\subsection{Comparisons and Controls}

\paragraph{Baselines and controls.}
We compare timing, tool/schema fingerprints, rare anchors, context overlap, and their multifeature
combination. Repository-only and context-only controls isolate direct namespace exposure and replay,
while random partitions establish the no-evidence baseline. Feature ablations remove each major
signal family in turn.

For \asl, a generic text-similarity baseline asks whether lexical resemblance alone explains the
result. We separately ablate replay, tool/resource continuity, typed handles, and conflict rejection.

Learned comparisons include dense semantic retrieval, approximate nearest-neighbor search, and
strict/cross-scaffold project matching. Together with supervised matching and filtering work
\cite{li2020ditto}, these comparisons place CARP within the broader entity-resolution design space.

\subsection{Evaluation Protocol}

\paragraph{Data separation.}
Every method receives the same standardized Open-SWE test view, while the full trajectories support
direct-exposure, longitudinal, and cost analyses. Development workflows remain disjoint from all
reported test metrics.

\paragraph{Calibration and uncertainty.}
All test decisions use only provider-visible observations. Open-SWE and historical tau use disjoint
development/test workflows, the hierarchy overlay fixes its vocabulary on early traffic, and profile or semantic
experiments split by entity. We bootstrap at the natural unit of each task; exact grids, resampling
counts, thresholds, budgets, and caps are in the supplement.

\paragraph{Paired transformations.}
Temporal stress preserves within-workflow order while compressing and jittering arrivals to create
progressively heavier concurrency. A lexical stress rephrases retransmitted tasks. The sanitized
Open-SWE project experiment calibrates on one project set and evaluates on unseen projects.

For sanitized Open-SWE, we rewrite provider-visible content to remove workspace/home paths,
repository fields and slugs, and internal domains before feature extraction. Hierarchy-overlay stress tests
independently vary anchor occurrence retention, later-traffic alias rotation, and shared-anchor
collision. The supplement specifies every transformation.

\subsection{RQ1: What Does Identifier Stripping Leave Exposed?}

Every request in the direct audit contains a parser-recoverable canonical repository and owner-like
label. Even when several project-like strings occur, the canonical namespace remains visible. This
establishes direct sanitization exposure before any cross-request inference.

Hybrid, \carp, and the repository-anchor control all recover nearly perfect cross-workflow project
and owner associations. Their agreement attributes the result to explicit OpenHands path components;
when SWE-agent omits that representation, the same anchor control yields no association.

After path abstraction, exact nesting first recovers sanitized sessions. Residual task, code, test,
and tool evidence then links previously unseen projects at F1 .682, while the direct-anchor stage
vanishes and dense-only matching remains weak. Explicit paths therefore dominate direct recovery,
but structured non-path evidence still supports substantial project association.

The same sanitized protocol transfers to SWE-agent. Target-calibrated matching reaches F1 .439; with
no target labels, mutual-nearest agreement preserves high precision and reaches F1 .318 from
OpenHands to SWE-agent and .179 in the reverse direction. The asymmetric recall shows that scaffold
representation changes the available evidence, while agreement-based edges remain reliable.

\subsection{RQ2: How Much Workflow Continuity Remains?}

Exact cumulative-prefix matching recovers every held-out Open-SWE workflow at F1 1.000, versus
Hybrid .983 and bounded \carp .967. On incremental-context requests \carp falls to .117; combining
non-cumulative requests with strict content removal lowers session F1 to .016 and eliminates
project/owner linkage. Exact nesting still solves sanitized cumulative requests, isolating replayed
dialogue from entity handles.

On historical tau workflows, time grouping degrades rapidly as concurrency rises, while the
time-independent content stage remains at F1 .960 (Figure~\ref{fig:overview}(B)). Rephrasing breaks
exact intent hashes, yet the content stage retains F1 .843, showing that semantic continuity survives
beyond literal replay.

\asl more than triples \carp's incremental-context F1 on Open-SWE (.394 versus .107) and improves
historical
tau linkage (.164 versus .104), while making no false session assertion in either corpus. It also
outperforms generic hashed-text linkage, especially on tau. Ablations attribute the Open-SWE gain to
tool/resource continuity, whereas cumulative tau requests combine replay and typed-handle evidence
(Figure~\ref{fig:overview}(C)).

\subsection{RQ3: Do Stable Anchors Enable Longitudinal Propagation?}

Natural tau traces show the same persistence outside the controlled overlay. Typed handles link
benchmark-user traffic across workflows at F1 .571; removing them eliminates the signal. In a later-
traffic split, alternate aliases recover nearly all assignments even after user IDs disappear. Thus
stable content handles recur across non-code workflows and support genuine longitudinal propagation.

The controlled hierarchy overlay isolates how cross-cache propagation and handle stability affect
coverage. Percolation lifts all three hierarchy levels from weak local clusters to F1 around .7--.8
while retaining high precision
(Figure~\ref{fig:t3}).

An early-traffic entity watchlist then assigns later traffic with F1 above .7 at every hierarchy
level and above .8 for projects and tenants. Thinning handle occurrences steadily reduces coverage;
complete alias rotation collapses the watchlist, and shared-anchor collisions trade recall for
precision. These
interventions show that persistence and uniqueness---not merely semantic topic---govern longitudinal
reach.

\paragraph{Partial technical profiling.}

Structured evidence raises technical-profile F1 from .623 to .809, essentially matching the score
obtained with truth clusters. Linkage error is therefore no longer the main bottleneck: once requests
are grouped, their content supports a stable partial view of technologies, services, and dependencies.

\subsection{RQ4: When Does Linkage Fail?}

With four requests/entity, the controlled expected F1 bound falls from 1.000 to .091 as exchangeable
entities/class rise from 1 to 16 (Figure~\ref{fig:overview}(D)). CARP reaches the bound for small classes
and abstains as ambiguity grows. Likewise, deliberately confounded user signals leave almost no
recoverable linkage, and full alias rotation makes watchlist F1 zero. The attack-independent limit is
complete visible-sequence exchangeability; shared prompts alone leave other distinguishing channels.

\subsection{RQ5: Can Risk Be Measured Under Bounded Pair Budgets?}

On two large overlays, streamed \carp reduces exhaustive comparison by roughly two orders of
magnitude. Sparse discovery also limits the semantic stage to a small post-discovery subset.

A scale experiment keeps comparisons per request essentially constant and retains every
true session candidate, while the reduction relative to all-pairs search grows linearly with traffic
volume. Linkage quality also remains stable under collision stress.

Across streams up to 500K requests, \asl sustains stable throughput and a fixed active candidate
working set.
Only the output partition state grows with the stream, matching the method's linear-space design.

On held-out Open-SWE, \carp's candidate union contains every true session pair, so its remaining
errors arise after blocking. A compact shingle-sketch alternative retains nearly all true candidates
and reaches strong final F1, confirming that approximate retrieval can preserve the linkage surface
at matched scale.

Candidate budgets and local overlap settings have little effect once true pairs enter the candidate
set. The main sensitivity is the required degree of context containment: stricter growth evidence
trades recall for precision.

A standard graph-based ANN index \cite{malkov2020hnsw} nearly matches exact dense retrieval on
tau-bench, showing that semantic continuity can also be served efficiently. Dense-only matching is
far weaker after Open-SWE path removal and under zero-label scaffold transfer. Learned
representations are therefore well suited to repeated non-code intent, whereas project linkage
depends more strongly on scaffold-specific structure.

\section{Mitigation Implications}

The controls require both handle minimization and less replay. A broker can translate customer or
repository names into request/session-scoped handles and forward only fields required by the current
tool. Stable pseudonyms preserve references but remain linkable; rotating handles complicate
cross-turn references, caching, debugging, audit joins, and exact account/order operations.

Non-cumulative transport need not erase memory: the broker can keep state and send summaries or
retrieved evidence. This may omit constraints, add latency, and reduce multi-turn quality. Utility
proxies expose a sharp tradeoff: session-scoped pseudonyms preserve almost all tool context but leave
substantial workflow linkage, whereas combining handle abstraction with aggressive context
minimization suppresses linkage at the cost of most tool evidence. The supplement quantifies this
privacy--utility frontier.

\section{Limitations and Ethics}

Our datasets combine public Agent trajectories with controlled entity overlays. Open-SWE provides
workflow, repository, and owner truth; tau provides benchmark-entity truth; and overlays provide
designed hierarchical truth, while enterprise prevalence remains unmeasured. The experiments isolate
content from network/account metadata, and the 500K replay evaluates computational scale. Consented
deployment studies can extend these results to prevalence and task-level defense utility; released
artifacts follow upstream licenses.

\section{Conclusion}

Removing IDs does not remove links. Protocol identifier stripping leaves three separable risks.
Persistent namespace and business-object handles expose longer-lived associations, while cumulative
prefixes expose workflow continuity. Even after paths and namespaces are removed, structured content
supports partial project linkage and rephrased-intent continuity; when handles persist, their stability
determines watchlist reach.
Conversely, strict path abstraction plus non-cumulative requests reduces session F1 from 0.967 to
0.016, showing that the channels require distinct controls. Together, these measurements locate the
