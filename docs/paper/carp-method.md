# CARP: Cache-Anchored Rarity Percolation

CARP is the paper-facing name for the scalable provider-side linkage method implemented by the
current `provider_lowcost` pipeline. The goal is to make the method contribution explicit:
instead of presenting a bag of known features, CARP is a cost-aware candidate-generation and
percolation algorithm under provider-visible constraints.

## Threat-Model Fit

CARP assumes a cold-start model API provider:

- The provider sees provider-visible Agent API request contents.
- The provider does not see broker-side `user_id`, `org_id`, `session_id`, `trace_id`, or
  evaluation labels.
- The provider can compute low-cost signals that naturally arise during inference or logging:
  cache-like buckets, tokenization, length, tool/schema fingerprints, rare structural tokens,
  and local context overlap.
- The provider avoids naive all-pairs semantic comparison.

## Core Idea

Agent requests can retain persistent content handles and replayed context after broker identifiers
are removed. CARP is a bounded reference attack that first
partitions traffic into cache-like buckets, then builds sparse candidate edges from rare
structural tokens and cumulative-context overlap. Connected components over these edges produce
workflow, user, project, and owner-like clusters.

The sparse-linkage layer has five deterministic steps:

1. **Cache anchoring.** Partition requests into coarse cache-like buckets.
2. **Rarity indexing.** Build inverted indexes over rare structural tokens within each bucket.
3. **Context-growth percolation.** Add edges consistent with cumulative context and temporal
   growth.
4. **Budgeted refinement.** Use bounded local comparisons to merge clusters when multiple weak
   signals agree.
5. **Strong-entity percolation.** For non-code traffic, parse tool JSON and explicit ID phrases,
   propagate level-appropriate user/process/tenant handles across cache buckets, keep shared
   resources candidate-only, and reject aliases that bridge stronger components.

## Algorithm Sketch

Input:

- provider-visible requests `R`
- feature budgets `B`, including text window, maximum shingles, maximum words, maximum candidate
  pairs/request
- edge thresholds `T`

Output:

- predicted clusters for `session`, `user`, `project`, and `org`
- runtime/candidate-pair statistics

Pseudo-code:

```text
CARP(R, B, T):
  buckets = group_by_cache_anchor(R)
  initialize union-find structures UF_session, UF_user, UF_project, UF_org

  for bucket in buckets:
      F = extract_budgeted_features(bucket, B)

      rare_edges = rarity_edges(F, max_bucket_size=B.rare_bucket_size)
      for (i, j, token_type, lift) in rare_edges:
          if token_type is trace/context:
              UF_session.union(i, j)
          if token_type is user-like:
              UF_user.union(i, j)
          if token_type is repo_full:
              UF_project.union(i, j)
          if token_type is repo_owner/domain:
              UF_org.union(i, j)

      context_edges = context_growth_edges(F, B.max_pairs_per_request)
      for (i, j) in context_edges:
          if context_score(i, j) >= T.context:
              UF_session.union(i, j)

      refinement_edges = local_agreement_edges(F, B.max_pairs_per_request)
      for (i, j) in refinement_edges:
          if score(i, j) >= T.session:
              UF_session.union(i, j)
          if score(i, j) >= T.user and user_signal_overlap(i, j):
              UF_user.union(i, j)
          if score(i, j) >= T.project and repo_overlap(i, j):
              UF_project.union(i, j)
          if score(i, j) >= T.org and owner_or_domain_overlap(i, j):
              UF_org.union(i, j)

  entity_graph = exact_business_entity_graph(R)
  reject ambiguous account-cache aliases
  percolate customer/order/queue/tenant components across cache buckets

  return clusters(UF_*), statistics
```

## Edge Score

CARP can be described as optimizing a sparse edge score:

```text
score(i, j) =
  alpha * rarity_lift(i, j)
+ beta  * context_growth(i, j)
+ gamma * tool_schema_match(i, j)
+ delta * time_length_compatibility(i, j)
+ eta   * local_semantic_agreement(i, j)
```

The implementation does not require dense scoring for all pairs. The score is evaluated only
for candidate pairs generated by cache anchors, inverted indexes, and bounded local refinement.

## Why This Is Not All-Pairs Matching

For `n` requests, naive all-pairs matching considers `n(n-1)/2` request pairs. CARP considers
only pairs that co-occur in small cache/rarity/context buckets.

The current 12k Dataset B runs illustrate the reduction:

- all-pairs for 12,000 requests: 71,994,000 pairs
- U3 streamed CARP/provider_lowcost candidates: 361,997 pairs
- U4 streamed CARP/provider_lowcost candidates: 366,211 pairs

This is roughly a 197x-199x candidate-pair reduction before any production-specific provider
optimizations.

## Paper Claim

Use this wording:

> CARP is a bounded reference attack for measuring provider-side Agent-log linkage. It turns
> provider-visible cache-like buckets, persistent typed handles, and context carryover into sparse
> linkage edges, enabling the three residual channels to be tested without naive all-pairs
> comparison.

Avoid this wording:

> We use rare tokens, schema fingerprints, timestamps, and semantic embeddings.

The latter sounds like a feature list; the former states a method.
