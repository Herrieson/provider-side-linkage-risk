# 给同学看的背景说明：为什么匿名的 LLM Agent 请求仍可能被重新关联

这份文档不是论文正文，也不关注实验细节。它的目标是让没有参与项目的同学快速理解：我们在研究什么问题、为什么这个问题重要、攻击者大概能做什么、我们的方法思路是什么。

## 1. 一句话解释这个工作

我们研究的是：

> 即使一个中间代理已经隐藏了用户 ID、组织 ID、session ID，模型 API 提供商仍可能通过 LLM Agent 请求内容本身，把匿名请求重新串起来，恢复出一个用户或项目的工作流，并逐渐形成技术画像。

这里的重点不是“某条 prompt 泄露了密码”，而是“很多看起来匿名的请求，放在一起之后可以被重新关联”。

## 2. 先理解场景：LLM Agent 是怎么调用模型的

普通 ChatGPT 对话通常是一问一答。用户问一个问题，模型回答一次。

但 LLM Agent 不一样。Agent 会为了完成一个任务，多次调用模型，并且每次调用可能带上很多上下文。例如一个代码修复 Agent 可能会：

1. 读取项目目录；
2. 查看某个文件；
3. 运行测试；
4. 把报错日志发给模型；
5. 修改代码；
6. 再运行测试；
7. 把新的结果继续发给模型。

所以一次 Agent 工作流里，会产生很多条 LLM API 请求。这些请求里可能包含：

- 当前 workspace 路径；
- GitHub 仓库名；
- 文件路径；
- 函数名、类名、模块名；
- 依赖包；
- 测试输出；
- shell 命令结果；
- CI 报错；
- 前几轮对话历史；
- 工具调用结果。

这些信息不一定是密码，也不一定是传统意义上的敏感信息，但它们可能非常“有辨识度”。

## 3. 为什么会有 broker 或 gateway

很多公司或平台不会让用户直接把请求发给模型 API 提供商，而是中间加一层 broker、gateway 或 proxy。

这层中间系统可以做很多事情：

- 隐藏真实用户 ID；
- 隐藏企业组织 ID；
- 隐藏 session ID；
- 做路由；
- 做权限控制；
- 做日志审计；
- 做脱敏；
- 把多个用户的请求混在一起再发给模型 API provider。

直觉上，如果模型提供商看不到用户 ID 和组织 ID，它应该就很难知道哪些请求来自同一个人、同一个项目或同一个公司。

但我们的工作想说明：这不一定成立。

## 4. 核心风险：请求内容本身也能当“指纹”

即使显式身份被删掉，请求内容里仍然可能有很多稳定线索。

举个简单例子。

假设模型 API 提供商看到下面几条匿名请求：

```text
请求 A：运行 pytest 后，/workspace/acme__billing-service/tests/test_invoice.py 报错。
请求 B：请修改 /workspace/acme__billing-service/billing/invoice.py 中的 round_tax 函数。
请求 C：新的错误来自 acme__billing-service 的 GitHub Actions job: billing-unit-tests。
```

这三条请求里没有 user ID，也没有 org ID。但它们都包含类似的路径、仓库名和测试信息。模型提供商很容易猜测：这些请求大概率属于同一个项目，甚至属于同一个 Agent 工作流。

再举一个不含明显仓库名的例子：

```text
请求 A：上一步工具输出显示 PaymentRetryPolicy 在第 42 行失败。
请求 B：继续分析 PaymentRetryPolicy 和 InvoiceBackfillJob 的关系。
请求 C：根据刚才的测试结果，InvoiceBackfillJob 仍然在边界日期失败。
```

这里即使没有完整路径，也有重复出现的类名、任务名和上下文。多条请求放在一起，也可能被关联。

所以问题不是单个字段泄露，而是很多弱信号叠加后形成了“内容侧指纹”。

## 5. 攻击者是谁

我们假设的攻击者不是黑客，也不是窃听网络的人。

攻击者是模型 API 提供商本身。

它是正常服务的一方，所以它本来就能看到用户发给模型的请求内容。它不需要入侵系统，也不需要拿到 broker 内部的用户表。它只需要在自己能看到的请求日志上做分析。

我们假设它一开始不知道：

- 哪个请求属于哪个用户；
- 哪个请求属于哪个公司；
- 哪些请求属于同一个 Agent session；
- 某个用户的历史画像。

这叫冷启动。

它要做的是：从一大堆匿名请求里，自己把相关请求找出来。

## 6. 它想恢复什么

我们把风险分成两步。

第一步是请求关联。

也就是从海量匿名请求中判断：

- 哪些请求属于同一个 Agent 工作流；
- 哪些请求属于同一个项目或仓库；
- 哪些请求可能属于同一个 GitHub owner 或组织近似实体；
- 在有用户标签的实验数据里，哪些请求属于同一个用户。

第二步是画像重建。

一旦模型提供商知道一组请求大概属于同一个工作流或项目，它就可以继续总结：

- 这个项目用什么语言；
- 用什么框架；
- 用什么测试工具；
- 用什么包管理器；
- 有没有 CI/CD；
- 有没有数据库、云服务、认证系统线索；
- 经常出现哪些内部服务名或模块名；
- 之后看到哪些 token 可以快速识别同一实体。

这就从“某条请求泄露一点信息”，变成了“长期积累一个项目或用户的技术画像”。

## 7. 方法大致怎么做

最朴素的办法是：把所有请求都做 embedding，然后两两比较相似度。

但这很贵。如果模型提供商每天有海量请求，全量两两比较不现实。

所以我们的思路是模拟一个更低成本的 provider-side 方法。

大致流程如下。

### 第一步：先用便宜信号粗分组

模型提供商在正常推理时，本来就可能知道或容易计算：

- 请求长度；
- token 数；
- 时间；
- prompt 前缀是否相似；
- cache 命中或类似 cache bucket；
- tool schema 是否类似。

这些信号不用来最终判断身份，只用来先缩小范围。

比如先把可能相似的请求放进同一个桶里。

### 第二步：在桶里找罕见 token

再看每个桶里有没有稳定重复的罕见 token，例如：

- workspace 路径；
- repo 名；
- 文件路径；
- 函数名；
- 类名；
- 包名；
- 错误字符串；
- CI job 名；
- tool 名。

如果两条请求共享一些很少见的 token，它们就可能有关联。

### 第三步：利用上下文递增

很多 Agent 请求是 cumulative context。也就是说，第二轮请求会带着第一轮的内容，第三轮又会带着前两轮的一部分内容。

所以同一个工作流里的请求，经常存在明显的上下文重叠和顺序关系。

这可以帮助恢复：

- 哪些请求属于同一个 session；
- 大概的 turn order；
- 一个 Agent 是怎么一步一步完成任务的。

### 第四步：再做轻量语义整理

前面几步已经把候选范围缩小了。最后可以用小模型、embedding 或简单规则做进一步合并和拆分。

重点是：不是一开始就对所有请求做昂贵的全量语义搜索，而是先用便宜信号把候选空间缩小。

## 8. 我们为什么不用一句话说“路径泄露”

这个工作里，路径确实是非常强的信号，尤其是对项目和仓库级别的重建。

例如 workspace path 或 repo slug 一旦出现，就很容易把请求关联到同一个项目。

但整个问题不只是路径泄露。

还包括：

- cumulative context；
- 代码符号；
- 工具输出；
- 罕见错误字符串；
- tool schema；
- 请求长度和时间；
- 多轮上下文的递增关系；
- 多个弱信号的组合。

所以更准确的说法是：

> LLM Agent 请求存在内容侧多信号关联面，其中 workspace/repo 路径是很强的一类信号，但不是唯一信号。

## 9. 数据为什么要分几类

真实数据很重要，但真实数据经常没有完整标签。

比如 Open-SWE 这类真实 Agent trace 可以告诉我们：

- 工作流能不能被恢复；
- 项目或仓库能不能被恢复；
- GitHub owner-like 实体能不能被恢复；
- 技术栈线索能不能被抽取。

但它没有可靠的真实 user ID。所以我们不能在 Open-SWE 上声称“恢复了真实用户”。

为了解决这个问题，我们用了几层数据：

### 真实 Agent Trace

用来证明真实请求内容里确实存在可关联信号。

### Open-SWE User Overlay

这是在真实 Open-SWE trace 基底上，人工注入受控 user-level 信息。它不是声称这些是真实用户，而是用来测试：

> 如果真实请求里存在类似的用户级稳定信号，方法能不能恢复？

### 合成数据

合成数据有完整真值，适合做机制验证，例如规模变大、难度变高、画像字段变多时会发生什么。

这三类数据回答的问题不一样，不能混在一起讲。

## 10. 这项工作的边界

这个工作不是说模型提供商一定能知道真实公司是谁。

更准确地说：

- 在真实 Open-SWE 数据上，我们能评估 workflow、project、GitHub owner-like linkage；
- Open-SWE 的 owner 不是企业组织；
- Open-SWE 没有真实 user ID，所以 user-level 不能在这个真实数据上直接评估；
- user-level 部分主要靠半合成 overlay 和合成数据做机制验证；
- 方法证明的是内容侧关联风险，不是说所有场景都会泄露同样强。

这些边界很重要。论文如果说得太满，反而不可信。

## 11. 为什么这个问题重要

因为很多人会以为：

> 我已经隐藏了用户 ID，所以 provider 看不到身份，隐私就安全了。

但对 LLM Agent 来说，身份不一定只存在于 ID 字段里。身份和环境可能散落在上下文里：

- 你的路径；
- 你的 repo；
- 你的文件命名；
- 你的错误日志；
- 你的依赖；
- 你的工具输出；
- 你的多轮上下文。

这些东西是 Agent 完成任务所需要的，但也可能让请求变得可追踪。

所以这个工作的意义是提醒系统设计者：

> 隐私防护不能只删显式身份字段，还要考虑内容侧准标识符和跨请求关联。

## 12. 防御大概应该怎么想

可能的防御方向包括：

- 尽量减少稳定 workspace/repo/path 暴露；
- 不要使用全局稳定的占位符，否则占位符本身也可能变成标识；
- 控制 cumulative context，把无关历史移除；
- 对 tool output 做最小化；
- broker mixing 不能只打乱时间，还要考虑内容信号；
- 在隐私和任务可用性之间做权衡。

最关键的是：不能只问“有没有 user ID”，还要问“请求内容能不能把同一用户或项目重新串起来”。

## 13. 最后用一个类比理解

可以把匿名 LLM Agent 请求想象成一堆没有署名的工作笔记。

每张纸上都没有写作者姓名，但纸上可能反复出现同一个项目名、文件路径、错误日志、代码风格、工具输出和任务进展。

如果有人长期收集这些纸，即使没有姓名，也可能把很多纸归到同一个人或同一个项目下面。

我们的工作研究的就是这个问题在 LLM Agent API 日志中的形式。
